# Gemini Docs Generation Skill

Google Gemini CLIを使用してプロジェクトドキュメントを自動生成するスキル。

## 概要

このスキルは、Geminiの強力なコンテキスト理解力を活用して、プロジェクトの各種ドキュメントを自動生成します。コードベース全体を分析し、適切で包括的なドキュメントを作成します。

## 実行コマンド

```bash
gemini "<request>" --files <file_patterns>
```

## パラメータ

| パラメータ | 説明 |
|-----------|------|
| `"<request>"` | ドキュメント生成の依頼内容（日本語可） |
| `--files` | 分析対象のファイルパターン |

## ドキュメントタイプ別の使用例

### 1. README生成
**目的**: プロジェクト概要、セットアップ手順、使用方法を含むREADMEの作成

```bash
gemini "このプロジェクトのREADME.mdを生成してください。以下の内容を含めてください: (1) プロジェクト概要、(2) 主要機能、(3) 技術スタック、(4) セットアップ手順、(5) 使用方法、(6) ディレクトリ構造、(7) コントリビューション方法" --files "**/*.{ts,tsx,js,jsx,json}" --files "CLAUDE.md"
```

**出力先**: `README.md`

### 2. API仕様書生成
**目的**: RESTful APIやGraphQL APIの仕様書を作成

```bash
gemini "API仕様書を生成してください。以下の内容を含めてください: (1) 全エンドポイント一覧、(2) リクエスト/レスポンス形式、(3) 認証方法、(4) エラーハンドリング、(5) レート制限、(6) サンプルコード" --files "src/api/**/*.{ts,js}" --files "src/routes/**/*.{ts,js}"
```

**出力先**: `docs/API.md`

### 3. ユーザーマニュアル生成
**目的**: エンドユーザー向けの使用方法ドキュメント

```bash
gemini "ユーザーマニュアルを生成してください。以下の内容を含めてください: (1) 基本的な使い方、(2) 各機能の詳細説明、(3) スクリーンショット推奨箇所、(4) FAQ、(5) トラブルシューティング" --files "src/components/**/*.{tsx,jsx}" --files "src/pages/**/*.{tsx,jsx}"
```

**出力先**: `docs/USER_MANUAL.md`

### 4. 開発者ドキュメント生成
**目的**: 開発者向けの技術ドキュメント

```bash
gemini "開発者ドキュメントを生成してください。以下の内容を含めてください: (1) アーキテクチャ概要、(2) コンポーネント設計、(3) データフロー、(4) 状態管理、(5) テスト戦略、(6) デプロイ手順" --files "**/*.{ts,tsx,js,jsx}" --files "CLAUDE.md"
```

**出力先**: `docs/DEVELOPMENT.md`

### 5. コンポーネントドキュメント生成
**目的**: UIコンポーネントのドキュメント（Storybook形式など）

```bash
gemini "コンポーネントドキュメントを生成してください。各コンポーネントについて: (1) 概要、(2) Props仕様、(3) 使用例、(4) バリエーション、(5) アクセシビリティ対応" --files "src/components/**/*.{tsx,jsx}"
```

**出力先**: `docs/COMPONENTS.md`

### 6. データベーススキーマドキュメント
**目的**: データベース設計のドキュメント

```bash
gemini "データベーススキーマドキュメントを生成してください。以下の内容を含めてください: (1) ER図の説明、(2) 各テーブルの詳細、(3) リレーション、(4) インデックス、(5) マイグレーション履歴" --files "prisma/schema.prisma" --files "src/db/**/*.{ts,js}"
```

**出力先**: `docs/DATABASE.md`

## 実行手順

1. **ドキュメントタイプの特定**: 生成するドキュメントの種類を確認
2. **関連ファイルの特定**: ドキュメント生成に必要なファイルパターンを決定
3. **Gemini実行**: 適切なプロンプトでGeminiを実行
4. **ドキュメント受け取り**: Geminiが生成したドキュメントを受け取る
5. **ファイル保存**: 適切な場所にドキュメントファイルを保存
6. **レビュー依頼**: 必要に応じてCodexによるドキュメントレビューを実行

## ドキュメント生成チェックリスト

### README
- [ ] プロジェクト概要の明確さ
- [ ] セットアップ手順の完全性
- [ ] 使用例の具体性
- [ ] バッジ（ビルドステータスなど）の追加

### API仕様書
- [ ] 全エンドポイントの網羅
- [ ] リクエスト/レスポンス例の提供
- [ ] 認証方法の説明
- [ ] エラーコード一覧

### ユーザーマニュアル
- [ ] 初心者でも理解できる説明
- [ ] ビジュアル要素の推奨箇所
- [ ] FAQ・トラブルシューティング
- [ ] 用語集

### 開発者ドキュメント
- [ ] アーキテクチャ図
- [ ] セットアップ手順
- [ ] コーディング規約
- [ ] CI/CD説明

## ドキュメントの品質向上

生成されたドキュメントは以下のプロセスで品質を向上させます：

1. **Gemini生成**: 初期ドキュメントの生成
2. **Codexレビュー**: `/codex-review`でドキュメントの品質確認
3. **改善**: レビューフィードバックを反映
4. **最終確認**: ユーザーによる最終確認

## 出力形式

生成されたドキュメントは以下のディレクトリ構造で保存されます：

```
/
├── README.md                    # プロジェクト概要
├── docs/
│   ├── API.md                  # API仕様書
│   ├── USER_MANUAL.md          # ユーザーマニュアル
│   ├── DEVELOPMENT.md          # 開発者ドキュメント
│   ├── COMPONENTS.md           # コンポーネントドキュメント
│   ├── DATABASE.md             # データベースドキュメント
│   └── ARCHITECTURE.md         # アーキテクチャドキュメント
```

## 注意事項

- Geminiは大量のファイルを一度に処理できるため、プロジェクト全体のコンテキストを理解したドキュメント生成が可能です
- 生成されたドキュメントは必ずレビューを行い、必要に応じて手動で調整してください
- 機密情報が含まれていないか確認してください
- ドキュメントは定期的に更新し、実装との整合性を保ってください

## ベストプラクティス

1. **段階的な生成**: 大規模なドキュメントは部分的に生成し、後で統合する
2. **テンプレート活用**: プロジェクト固有のテンプレートがある場合は事前に指定する
3. **コード例の検証**: 生成されたコード例が実際に動作するか確認する
4. **定期更新**: コードベースの変更に合わせてドキュメントを再生成する
